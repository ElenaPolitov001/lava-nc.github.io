class lava.core.generic.process.AbstractProcess(*args, **kwargs)¶

Abstract base class for any type of process.

An AbstractProcess represents a computational graph composed of variables and operations and may expose an application-specific user interface to manage process behavior. In order to execute a process, it must be compiled for a particular backend. A backend might be a neuromorphic hardware system or a conventional CPU/GPU platform. There is a compiler for each supported backend. During compilation, the compiler creates an executable for a process which in turn gets executed by an executor. The executable may hold runtime-related state and specify the runtime behavior of a process while the executor provides a backend-agnostic interface to the process and executable and manages the runtime execution of a process.

Processes define the state variables of a computational graph while an executable defines the behavior of the graph. Behavior and state are separated because the behavior might be realized in different ways depending on the backend while the public state variables are port of the universal API of a process.

Processes are nested/hierarchical structures. That means processes can either define a computational graph directly through variables and operations or be composed of other processes. This results in a tree-like hierarchy of processes.

There are different fundamental types of processes:

  • LeafProcess is the base type of any process than defines a computational graph of variables and operations but does not contain other processes.

  • Process is the base type of a hierarchical process which is only allowed to contain other Process or LeafProcess instances.

  • NeuralProcess is the base type for any process executable on neuromorphic cores.

  • SequentialProcess is the base class for any process which is only executable on conventional CPU/GPUs that process code sequentially.

  • AbstractCompartments is the base type of any process that represents the state and behavior of neural compartments.

  • AbstractConnections is the base type of any process that represents the state and behavior of neural connections.

How to define custom processes

To define a new process, developers must inherit from any of AbstractCompartments, AbstractConnections, SequentialProcess or Process and implement at least the _build(..) method. Within the _build(..) method, one can allocate all of a process’s internal or input/output variables and sub processes as well as use set_backend_executable_class(..) to define one or multiple executable classes that are used to execute a process on one or multiple backends.

In addition to the allocation of variables and sub processes, some backends may require the implementation of the process behavior in terms of conventional sequential code which is done in the respective executable. Depending on the process type, there are, however, some differences required in the definition of the executables:

  • SequentialProcesses only execute on CPU/GPUs and therefore always require a behavioral definition as part of the registered executable.

  • NeuralProcesses on the other hand may also execute on neuro cores on a neuromorphic backend which implement the computational graph directly in hardware and therefore only require a behavior definition if one would like to also execute them on a CPU/GPU backend. However, for processes executing on neuro cores, the respective executable must also provide the logic to compile a process onto a neuro core.

The Python modules for these different backends go into more details on how to implement executables for different backends. The bottom line is that the definition of a custom process must always be accompanied by the definition of one of multiple executables for a specific backend.

Connecting processes

Processes contain internal as well as interface variables through which processes can be connected with each other to exchange data. Processes or their variables get connected by assigning variables of one process to variables of another process.

This assignment of variables is accomplished by ‘calling’ a process as in this example:

p1 = Process1(..<in_args>)..)
p2 = Process2(..<in_args>..)
p2(input_a=p1.output_a, input_b=p1.output_b)

In this example, input_a, input_b, output_a and output_b must be legal variables of processes Process1 and Process2.

Of the different types of variables, it is only possible to connect output variables to input variables and internal variables to reference variables.

For more details on variables, see the corresponding documentation.

run(num_steps: int, backend: Optional[lava.core.generic.enums.Backend] = None, **kwargs)¶

Runs process on given backend for given number of time steps.

run(..) can be called multiple times in a row. The specification of backend is only required during the first call. If the process has not been compiled yet, it will be automatically compiled for the given backend.

This method is only an alias for the executor of the run(..) method of the given backend. Check documentation of underlying executor’s run(..) method for more optional input arguments.

Parameters

  • num_steps (int) – Number of time steps for which to execute the process.

  • backend (Backend) – Backend on which to execute process.
